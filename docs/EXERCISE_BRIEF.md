# GoCalma Exercise Brief

This document captures the challenge statement, delivery targets, and operating constraints for the GoCalma project.

## Challenge

**Track:** Privacy & Open Source AI Tools

### Problem Description

People regularly upload sensitive PDFs such as pension statements, social security notices, insurance forms, invoices, and tax documents to cloud AI tools. Once those files leave the user device, the contained personal data is no longer under the user's control.

GoCalma exists to provide a safe alternative: redact personal information locally before any document is shared with AI systems or third-party services.

### Primary Objective

Build an open-source tool that:

- accepts PDF documents locally,
- detects and redacts personal information,
- produces a safe redacted PDF for downstream AI usage,
- supports reversible redaction through a user-held key that maps placeholders back to original values.

### Super Challenge

Also redact sensitive information contained in scanned documents and embedded images.

## Resources And Suggested Tooling

### Datasets

- Synthetic sample PDFs such as mock invoices, salary statements, insurance letters, and tax forms
- Additional synthetic test data generated by participants
- Real personal data must not be used

### Suggested Open-Source Components

- [Microsoft Presidio](https://microsoft.github.io/presidio/) for local PII detection
- [Hugging Face](https://huggingface.co/) models and Transformers.js for local NER
- `pdf.js` or `pdf-lib` for PDF parsing and manipulation
- `tesseract.js` for OCR and image-based redaction

## Deliverables

Expected final prototype deliverables for April 2, 2026:

- Public open-source repository
- Working demo that accepts a PDF, detects PII, redacts it, and exports a redacted PDF
- Encrypted key file or wallet-compatible export for un-redaction
- README with setup instructions, architecture overview, and usage examples

## Judging And Success Metrics

- PII detection recall of at least 90 percent on test PDFs
- Zero PII leakage outside the local device
- Exact un-redaction fidelity using the user's local key or wallet export
- Workflow usability for non-technical users

## Ethics And Compliance

### Ethical Guidelines

- Privacy by design is mandatory
- No personal data may be processed outside the local device by default
- Redaction keys must remain local and never be transmitted
- The tool should expose a review step before finalizing redaction

### Legal And Compliance Expectations

- No plain personal data stored on servers or sent to third parties
- Only compatible open-source dependencies should be used
- Development and evaluation must use synthetic data
- Real documents belonging to other people must not be used

## FAQ

### What is the expected deliverable?

A simple redaction system that allows users to:

- upload a PDF,
- detect personal information automatically,
- review the detected fields,
- download the redacted PDF.

### Does the LLM need to run locally?

Yes for the core workflow. Local execution is required to guarantee that personal data does not leave the device. An optional external-provider path can exist only as an explicit user opt-in.

### Recommended build order

1. Implement text redaction for text-based PDFs.
2. Extend the system to image and scanned-document redaction.

If the input is image-based, the output should remain image-based with sensitive regions redacted.

### Which languages matter most?

Prioritize left-to-right European languages:

- English
- German
- French
- Italian
- Spanish

### What counts as personal information?

Examples include:

- full names,
- addresses,
- phone numbers,
- email addresses,
- national identification numbers,
- AHV / AVS numbers,
- social security numbers,
- passport numbers,
- patient IDs,
- insurance numbers,
- bank account numbers,
- IBANs,
- credit card numbers.

Context matters. A first name alone may not identify someone, while a first name combined with an address often does.

### Why build this instead of using existing tools?

Most current redaction tools are either too specialized, too hard to use, or rely on cloud processing. This exercise is specifically about building something that is:

- simple,
- privacy-first,
- open source,
- flexible across document types.

## Project Direction For This Repository

For this repository, the current working direction is:

- local-first PDF redaction is the core submission path,
- reversible token mapping is mandatory,
- OCR and image redaction are stretch goals,
- recall and reviewability matter more than silent misses,
- no implementation should weaken the local privacy guarantee.
